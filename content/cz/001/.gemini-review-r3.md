## Workflow Assessment

### What worked well

1. **Direct editing** — reading files → Gemini → Edit tool is much cleaner than the old extract→pipe→parse→apply cycle
2. **Two-pass structure justified** — Pass 1 caught calques and word order issues; Pass 2 caught semantic misses (e.g., "chyby v místech", "pomstít se")
3. **Foreground Gemini calls** work reliably with temp files and piped stdin
4. **Batch sizing** (5-8 entries for short texts) hit the sweet spot

### What could improve

1. **Background Gemini fails** — the CLI needs a TTY for auth; `run_in_background` breaks it. All calls must be foreground, which means sequential processing
2. **Rate limit pacing** — `sleep 5` between batches was sufficient for 8 calls but may need tuning for larger carnets
3. **Pass 2 self-confirmation** — despite the prompt instruction to "disregard GEM corrections", Pass 2 sometimes referenced or reversed Pass 1 fixes. Could strengthen the prompt or strip GEM comments before Pass 2
4. **No review files for C-severity** — the new workflow discards these entirely. If someone later wants to review what Gemini suggested but we skipped, there's no record. Could optionally save raw output

### Comparison with Round 2 (old workflow)

Round 2 (2026-02-14) used the old extract→structured-output→parse approach. The fixes overlap significantly, suggesting the new direct-edit workflow catches the same issues with less complexity. The main difference: Round 3 applies fixes inline with GEM comments, while Round 2 saved raw Gemini output as separate review files.

Nothing from this review warranted propagation to `content/_original/`. All issues were Czech-specific (calques, clitic placement, vocabulary).
